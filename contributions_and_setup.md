# Quantum Reinforcement Learning — Contributions & Setup

## 1. Technical Breakdown of Contributions

### A. Upgrading the Quantum Architecture to a QCNN

Currently, the actor and critic networks in the repo use a standard Variational Quantum Circuit (VQC) with generic alternating layers of $R_y$ rotations and CNOT entangling gates.

**What needs to be done:** Rip out the generic $U(\theta)$ block and replace it with a Quantum Convolutional Neural Network (QCNN). QCNNs mimic classical CNNs by using hierarchical feature extraction.

**The Architecture:** Define a circuit that applies a parameterized 2-qubit unitary across adjacent qubits (the convolutional layer), followed by a pooling layer where you either measure and discard qubits or apply controlled rotations to reduce the dimensionality.

**Implementation Details:** If the repo uses Qiskit or PennyLane, this is very modular. In PennyLane, for example, you can build custom convolutional blocks using `qml.StronglyEntanglingLayers` mapped across subset wires, iteratively halving the active qubits until you are left with the final qubits for your Pauli-Z measurement. This is highly effective for time-series data, as the convolution captures local temporal correlations in the asset prices before entangling the global state.

---

### B. Modernizing the Forecaster (Deep Learning)

The environment's state vector currently looks like this:

$$s_t = [p_{t-L:t},\ \hat{p}_{t+1:t+F+1}]$$

where $\hat{p}$ is a 7-day forecast generated by a classical Auto ARIMA model.

**What needs to be done:** ARIMA is computationally expensive and struggles with highly non-linear market regimes. Replace the ARIMA module with a lightweight PyTorch deep learning model, such as a 1D-CNN or an LSTM.

**The Architecture:** Build a separate, small neural network that takes the $L$-day lookback window and outputs an $F$-day forecast.

**Implementation Details:** Train this classical deep learning forecaster offline using the historical dataset. Once trained, freeze the weights and integrate the model into the RL environment's `step()` function. Instead of calling the ARIMA solver at every time step, run a fast forward-pass of your PyTorch model to generate $\hat{p}$ and append it to the state representation.

---

### C. Implementing Angle Encoding

The authors use amplitude encoding to compress the data. While this saves qubits (requiring only $\mathcal{O}(\log_2(N))$ qubits for $N$ features), it results in a very deep circuit with a gate complexity of $\mathcal{O}(N)$.

**What needs to be done:** Add a feature flag to swap the data embedding strategy from amplitude encoding to angle encoding. Angle encoding maps each classical feature directly to the rotation angle of a single qubit.

**The Architecture:** Instead of building a complex state-preparation circuit, apply parallel $R_y$ or $R_x$ gates to an initialized $|0\rangle^{\otimes N}$ state.

**Implementation Details:** This will require increasing the number of qubits in the circuit to match your feature dimension. If the feature space is too large for the available qubits, add a classical dimensionality reduction step (like PCA or a small Autoencoder) before the quantum embedding layer. This is an excellent experiment to benchmark the trade-off between qubit count (hardware constraints) and circuit depth (noise constraints).

## 2. How to Swap the Dataset (Cryptocurrency Example)

Swapping traditional financial assets for a cryptocurrency dataset is a highly practical experiment. The core QRL algorithms don't need to be rewritten, but the data ingestion and preprocessing pipelines will need adjustment.

### Step 1: Data Ingestion

Pull historical daily closing prices for a basket of cryptocurrencies (e.g., BTC, ETH, SOL) using libraries like `yfinance` or the Binance API. Output this data as a CSV or Pandas DataFrame that matches the exact column structure of the original repository's dataset.

### Step 2: Preprocessing Adjustments

The preprocessing module prepares the state for the quantum agent. Check the following carefully:

- **Time Series Continuity:** Traditional markets trade 5 days a week. Crypto trades 24/7. If the environment's logic hardcodes assumptions about business days (e.g., skipping weekends for rolling windows), update the indexing logic to account for a continuous 7-day week.

- **Log Returns:** Ensure the pipeline correctly calculates the logarithmic differences of the prices before feeding them into the normalization function.

- **Feature Map Normalization:** The repo uses a specific feature map $Z(\tilde{x})$ that scales the input and expands it with polynomial and trigonometric terms to break radial symmetry. Ensure crypto data passes through this exact transformation before hitting the quantum embedding.

### Step 3: Environment Constraints

RL environments in finance are sensitive to transaction costs. The original paper evaluates performance with an assumed transaction cost of **0.15% per trade** (typical for institutional stock trading). Crypto exchanges usually have different fee structures (e.g., 0.10% on Binance).

Locate the reward calculation in the Gym environment — where it deducts fees based on weight changes $|w_{t,i} - w_{t-1,i}|$ — and update the scalar to reflect realistic crypto trading fees.

## How to Run

1. Download and extract the ZIP file.
2. Create a virtual environment with **Python 3.12.6**.
3. Install dependencies:
   ```bash
   pip install -r requirements.txt
   ```
4. Run the main script:
   ```bash
   python MAIN.py
   ```

> **Note:** The code will take significant time to run — it uses 2 to 3 different quantum methods and can take **an hour or more**. To speed things up, comment out entries in the `MODELS_TO_RUN` hyperparameter.
